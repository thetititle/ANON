## 1. 프로젝트 개요 (지난 기획서 대비 업데이트 사항)

지난 기획서에서 제시한 *디지털 추모 플랫폼*의 큰 방향성은 유지하되, 컨셉의 정서적 깊이와 기술 스택의 구체성을 보강하는 방향으로 다음과 같이 업데이트하였습니다.

### 1.1 프로젝트명 및 철학 변경

| 구분 | 지난 기획서 | 업데이트 |
|---|---|---|
| 프로젝트명 | **고요 (GOYO)** | **안온 (安溫)** |
| 핵심 철학 | 일상과 죽음을 잇는 디지털 공간 | + **'디지털 천도(昇天)'** 개념 추가 — 한국 전통 49제를 기술로 재해석 |
| 정서적 컨셉 | 품격 있는 추억 | **슬픔의 완화와 수용** 중심의 심리적 완충지대로 구체화 |
| 서비스 역할 | 디지털 추모 공간 | **온라인 부고장 + 디지털 추모 공간** 이중 역할 — 부고 공유 → 조문 메시지 → 디지털 앨범 열람까지 단일 링크로 처리 |

### 1.2 신규 추가된 핵심 기능

- **49제 천도 시스템**: 입력된 기일로부터 49일째를 자동 계산하여, 해당 시점부터 서비스 전체 테마가 *'빛과 평온함'* 톤으로 전환되는 천도 연출 기능 신설
- **법적 대응 가능한 아카이브**: 추모 메시지 소프트 삭제 + 별도 아카이브 테이블(`memorial_comments_archive`)에 `deleted_at` / `deleted_by`까지 보존 — 고인 모독 발생 시 법적 증거 자료로 활용 가능한 구조 설계
- **RLS(Row-Level Security) 기반 본인 인증**: 작성자 외 수정 / 삭제를 DB 수준에서 원천 차단
- **게스트 / 인증 조문 하이브리드 모델**: 로그인한 유족은 `user_id`로, 비로그인 조문객은 `guest_name` + `guest_phone`으로 식별하는 이중 트래킹 구조 설계 — 폭넓은 조문 참여를 허용하면서도 추적 가능한 연락처를 확보하는 구조 (향후 SMS 인증 연동으로 신원 확인 강화 예정)
- **공개 / 비공개 추모 토글**: `memorials.is_public` 플래그를 통해 유족이 추모 공간의 외부 공개 여부를 직접 통제
- **부의금 범용 QR 코드**: 무빈소 장례 환경 대응을 위한 계좌 정보(`bank_name` / `account_number` / `account_holder`) 보관 및 QR 생성 기능 신설
- **온라인 부고장 + 조문 메시지 시스템**: 생성된 추모 페이지(`/memorial/[id]`)가 온라인 부고장 역할을 겸하도록 설계 — 유족이 링크를 공유하면 지인이 별도 앱 설치 없이 접속하여 고인에게 메시지를 남기고 다른 조문객의 메시지를 함께 열람할 수 있음. 인증 사용자와 비로그인 게스트 모두 조문 참여 가능 (`memorial_comments` 테이블 활용)
- **디지털 앨범**: 유족이 업로드한 고인의 생전 사진·영상을 추모 페이지에서 Swiper.js 기반 가로 슬라이더로 열람 — 최대 이미지 20장 + 영상 1개, Supabase Storage CDN 경유 제공 (`memorial_media` 테이블 + `order` 컬럼으로 순서 보존)

### 1.3 기술 스택 구체화

| 구분 | 지난 기획서 | 업데이트 |
|---|---|---|
| Frontend | React.js + PWA | **Next.js 16.2.4 + React 19.2.4** (App Router) |
| 스타일링 | (미명시) | **CSS Modules + Pretendard Variable**, 3단계 반응형(PC / 태블릿 / 모바일), 다크모드 대응 |
| 미디어 | (미명시) | **Swiper.js** 기반 가로 슬라이더, 고인당 이미지 20장 + 영상 1개 제한, **`Promise.all` 기반 이미지 병렬 업로드**로 Storage 전송 지연 최소화 |
| 인증 | 소셜 통합 인증 | 동일 (카카오 / 네이버 / 구글 OAuth 2.0), **실명 정보 스코프 승인 정책 명문화** |

### 1.4 UX 흐름 구체화

지난 기획서에서 *"챗봇 기반 온보딩"* 으로만 정의됐던 등록 절차를 **사용자 심리 상태를 고려한 3단계, 총 8스텝**의 점진적 입력 흐름으로 구체화하였습니다.

1. **현실 인지 단계** — 관계 설정, 기일 입력
2. **정서적 연결 단계** — 호칭 입력, 성격 키워드 회상
3. **기록 및 현실적 정리 단계** — 추억 미디어 업로드, 부의금 / QR 생성

### 2.2 4월 — 데이터 모델링 및 핵심 기능 (완료)

- **Supabase 프로젝트 생성 및 5개 테이블 정규화 설계 완료**

  | 테이블 | 역할 | 주요 설계 포인트 |
  |---|---|---|
  | `users` | 실명 인증 사용자 정보 | `auth.users` 외래키 연동, `real_name` NOT NULL로 실명 필수화 |
  | `memorials` | 고인 추모 페이지 본체 | 기일·관계·호칭·성격 태그(`text[]`)·공개 여부·부의금 정보 통합 |
  | `memorial_media` | 추모 미디어 | `type` CHECK 제약(`image` / `video`), `order` 컬럼으로 슬라이더 순서 보존 |
  | `memorial_comments` | 조문 메시지 | `user_id` nullable + `guest_name` / `guest_phone` 병기 → 인증 / 게스트 하이브리드 |
  | `memorial_comments_archive` | 삭제 메시지 보존 | `original_id` / `deleted_at` / `deleted_by` 보존, 법적 증거 자료 활용 |

- **외래키 cascade 정책 설계** — 사용자 탈퇴 시 본인 추모 페이지는 `cascade`, 조문 메시지의 `user_id`는 `set null`로 처리하여 *조문 흔적은 유지하되 작성자 식별만 분리*
- **추가 예정 컬럼** — `memorials` 테이블에 고인의 **생년월일(`birth_at`) 컬럼 추가 작업 예정**
  - **현재 용도**: 추모 페이지 상단의 *생존기간*(예: `1950 ~ 2026`) 표시
  - **향후 확장 대비**: 본 프로젝트의 최종 목표인 *의료기관 연계형 사후 케어 솔루션*으로 발전할 경우, 사망신고 절차 자동화에 고인의 생년월일 데이터가 법적 필수 항목이 됨. 이를 사전에 확보해 두는 의도 포함
  - **현재 상태**: 폼 스텝에서 입력은 받고 있으나 저장 경로 미연결, 5월 잔여 작업으로 이관
- **공통 컴포넌트 7종 개발 완료**

  | 컴포넌트 | 역할 |
  |---|---|
  | `ChatBubble` | 챗 UI 좌 / 우 정렬 메시지 |
  | `ProgressBar` | 폼 진행률 표시 |
  | `FormInput` | Enter 키 완료, sizer 기반 자동 너비 |
  | `DateInput` | 숫자 자동 포커스 이동 + 확인 alert |
  | `ChipSelect` | Swiper 가로 스크롤, 최대 선택 제한 |
  | `CustomAlert` | backdrop + primary / secondary 버튼 |
  | `MediaUpload` | 이미지 20장 + 영상 1개 썸네일 그리드 |

- **8단계 폼 스텝 정의 완료**: `name → birth → death → relation → media → personality → nickname → condolence`

## 5. 개발 도구 활용

투명성 확보 및 1인 개발 파이프라인의 효율성 검증 차원에서, 본 프로젝트 진행 과정에서 AI 보조 도구(Claude Code)를 활용하였음을 명시합니다.

### 5.1 AI 도구가 보조한 부분 (구현 보조)

- React 컴포넌트 보일러플레이트 작성 보조
- TypeScript 타입 정의 보조
- CSS Modules 작성 및 반응형 스타일 작성 보조
- 챗 UI 순차 공개 / 자동 스크롤 등 인터랙션 로직 디버깅
- 코드 리팩토링 제안 및 검토

### 5.2 본인이 직접 결정 · 수행한 부분 (기획 및 설계)

- **프로젝트 컨셉 정립** — '안온' 명명, 49제 디지털 천도 개념 도출
- **사용자 경험 흐름 설계** — 3단계(현실 인지 / 정서적 연결 / 기록 및 정리) UX 구조 설계 및 8스텝 폼 정의
- **디자인 시스템 토큰 정의** — 색상 / 타이포그래피 / 라운드 / 반응형 rem 등 토큰 체계 수립
- **데이터베이스 스키마 설계** — `users` / `memorials` / `memorial_media` / `memorial_comments` / `memorial_comments_archive` 5개 테이블 정규화 구조, 외래키 cascade 정책, 게스트·인증 하이브리드 조문 모델, 소프트 삭제 아카이브 정책까지 직접 설계
- **법적 대응 아카이브 구조 설계** — 소프트 삭제 + DB 트리거 기반 보존 정책
- **RLS 보안 정책 설계** — 작성자 본인 인증 기반 수정 / 삭제 권한 모델
- **의료기관 연계 비즈니스 전략 수립** — 대학병원 원무 시스템 연동 및 사후 케어 파이프라인 기획 (졸업 이후 확장 로드맵으로 설계)
- **49제 천도 연출 컨셉 기획** — 빛 테마 전환의 정서적 의미 설계
- **무빈소 장례 대응 기능 기획** — 부의금 범용 QR 등 현실적 지원 기능 정의

본 프로젝트의 *기획 의도, 정서적 컨셉, 시스템 설계*는 모두 본인이 직접 도출하였으며, AI 도구는 결정된 설계를 코드로 구현하는 단계에서의 생산성 보조 역할로 한정하여 활용하였습니다.
